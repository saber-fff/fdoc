## 1. 项目背景

### 1.1 现状概述

源启科技当前业务管理存在以下痛点：
- **系统割裂**: 交付管理在伙伴云，研发管理在腾讯表格，财务销售在用友系统，各系统间数据未打通
- **流程不透明**: 需求、任务、交付状态分散管理，缺乏统一视图
- **成本核算困难**: 缺乏标准化的产品清单和WBS，无法准确核算项目成本
- **协同效率低**: 跨部门协调机制不完善，存在"踢皮球"现象

### 1.2 建设目标

构建基于伙伴云的一体化管理平台，实现：
- **数据贯通**: 打通交付、研发、财务、销售全流程数据
- **标准化管理**: 建立产品元件、WBS、人员能力等标准化体系
- **成本可控**: 实现项目预算、核算、结算的闭环管理
- **决策支持**: 为管理层提供经营看板和决策依据

### 2.2 系统边界

| 系统 | 负责模块 | 数据流向 |
|------|---------|---------|
| **伙伴云** | 研交一体化管理(项目+研发+质量)、运维管理、人事考核 | 主数据管理，产品研发项目回写用友 |
| **用友** | 财务管理、营销管理(销售/客户/合同)、采购管理、库存管理 | 主数据同步，成本核算 |
| **企业微信** | 消息通知、文件存储 | 触发通知，文档归集 |
| **代码仓库** | 代码管理、CI/CD | 版本状态同步 |

**特殊说明**：
- **营销管理在用友**：商机、客户、合同等营销业务在用友系统管理，通过API与伙伴云交互
- **产品研发一体化**：各业务事业部实行"一部两制"，既负责产品优化迭代（研发部），也负责项目交付（开发部）。产品研发项目在伙伴云立项时即回写用友，实现项目全生命周期管理与财务核算一体化

## 4. 各模块详细设计

### 4.1 主数据管理模块（伙伴云）

#### 4.1.1 组织架构管理

**建设诉求**：支持2026年事业部制与销售事业部(阿米巴)并存的组织架构，实现人员单归属、项目双标记

**建设内容**：
- 业务单元管理（事业部/销售事业部(阿米巴)）
- 部门信息管理
- 人员信息管理（bu_id、bu_type）
- 薪酬成本管理
- 能力资质管理

**业务流程**：

```
人员归属流程：
人员入职 → 分配业务单元(bu_id) → 标记类型(bu_type:事业部/销售事业部) 
    → 归属部门(dept_id) → 分配职级和工时成本

成本归集流程：
人员填报工时 → 关联任务 → 汇总WBS → 归集项目成本 
    → 按bu_id归集到事业部/销售事业部
```

**数据交互**：

| 交互方向 | 数据内容 | 触发时机 | 说明 |
|---------|---------|---------|------|
| 伙伴云→用友 | 人员成本、工时 | 每日/周 | 人工成本分摊数据 |
| 用友→伙伴云 | 成本核算结果 | 月末 | 实际成本回流 |

---

#### 4.1.2 客户台账管理

**建设诉求**：建立统一客户主数据，支持交叉销售和成本标准管理

**建设内容**：
- 客户主信息管理
- 客户联系人管理
- 信息化情况管理
- 客户分级管理（ABC分级）
- 成本预算标准
- 合同外功能记录

**业务流程**：

```
客户信息录入流程：
客户信息录入 → 区域分配 → 客户分级评定 → 成本标准设定 
    → 销售人员分配

客户分级评定流程：
收集客户数据 → 计算客户价值 → ABC分级 → 服务等级分配
```

**数据交互**：

| 交互方向 | 数据内容 | 触发时机 | 说明 |
|---------|---------|---------|------|
| 用友→伙伴云 | 客户基本信息 | 合同签订 | 新客户同步 |
| 伙伴云→用友 | 客户信息化情况 | 定期同步 | 用于交叉销售分析 |

#### 4.1.3 产品/服务仓库

**建设诉求**：建立标准化的产品元件、WBS任务池，支持版本管理和成本核算

**建设内容**：
- 产品元件管理（最小功能单元）
- 产品版本管理（多版本共存）
- WBS任务管理（P1-P4工时标准）
- 版本-WBS关系管理
- 可售产品套餐（SPT）
- 销售方案模板

**业务流程**：

```
产品元件定义流程：
产线定义元件 → 拆分WBS任务 → 定义P1-P4工时标准 → 关联实施文档 
    → 形成WBS任务池

版本创建流程：
创建产品版本 → 从WBS任务池选择任务 → 设置版本特定工时 
    → 标记必选/可选任务 → 版本发布

销售套餐配置流程：
创建销售套餐(SPT) → 选择产品元件组合 → 配置产品版本 
    → 生成可售套餐
```

**数据模型关系**：

```
产品元件(prd_component)
    ├── 1:N 产品版本(prd_component_version)
    │         └── N:M 版本-WBS关系(prd_version_wbs)
    │                   └── M:1 WBS任务(prd_wbs_task)
    └── 1:N WBS任务(prd_wbs_task)

销售套餐(prd_package)
    └── 1:N 套餐明细(prd_package_item)
              └── N:1 产品元件(prd_component)
```

**数据交互**：

| 交互方向 | 数据内容 | 触发时机 | 说明 |
|---------|---------|---------|------|
| 伙伴云→用友 | 产品成本标准 | 定期同步 | 用于财务预算 |
| 用友→伙伴云 | 产品销售价格 | 定期同步 | 用于报价参考 |

```
           │         └──→ 经营生产项目

           └──→ 销售订单

    (伙伴云)              工时实时同步      (财务管控)
```

### 4.3 研交一体化管理模块（伙伴云-本次重点）

研交一体化管理将项目交付、研发过程和质量管控整合为统一的管理体系，实现从需求到交付的端到端闭环管理。

#### 4.3.1 项目管理

项目管理子模块负责项目全生命周期的交付管理，包括经营生产项目（客户交付类）和产品研发项目（产品迭代类）两大类。

**项目类型说明**：

| 项目类型 | 定义 | 负责方 | 回写用友 |
|---------|------|--------|---------|
| **经营生产项目** | 面向客户的交付项目 | 各事业部交付团队 | ❌ 在用友立项 |
| **产品研发项目** | 各事业部自身产品优化迭代 | 各事业部研发团队 | ✅ 完成后回写 |

**各业务事业部产品管理**：
- 大健康运营事业部：负责大健康产品线迭代
- 标准化项目事业部：负责标准化产品迭代  
- 大项目事业部：负责大项目定制开发
- 效能与品质保障事业部：负责平台工具和效能产品迭代

各事业部在伙伴云维护自身的产品研发项目，完成后回写用友进行财务核算。

##### 4.3.1.1 项目启动

**建设诉求**：基于合同快速立项，组建项目团队，实现项目双标记

**建设内容**：
- **经营生产项目**：接收用友立项信息，面向客户交付
- **产品研发项目**：各事业部自主创建，产品优化迭代，完成后回写用友
- 团队组建
- 项目双标记（bu_id + amoeba_id）
- 项目基础信息维护

**业务流程**：

```
【经营生产项目立项流程】
接收用友立项信息 → 创建项目台账 → 标记执行方(bu_id) 
    → 标记销售方(amoeba_id) → 关联客户和产品清单 → 组建项目团队

【产品研发项目立项流程】
各事业部创建产品研发项目 → 标记所属事业部(bu_id) 
    → 关联产品元件 → 组建研发团队 → 研发完成后回写用友

项目双标记规则：
• bu_id（执行方）：项目交付归属的事业部
• amoeba_id（销售方）：项目销售归属的阿米巴
• region_code（区域）：项目所属地理区域

示例：某三甲医院项目
• 销售方：经营一部(amoeba_id=经营一部)
• 执行方：标准化项目事业部(bu_id=标准化项目事业部)
```

**数据交互**：

| 交互方向 | 数据内容 | 触发时机 | 说明 |
|---------|---------|---------|------|
| 用友→伙伴云 | 立项编号、合同信息 | 合同签订 | 项目创建依据 |
| 用友→伙伴云 | 财务立项、预算 | 项目立项 | 预算控制依据 |
| 伙伴云→用友 | 项目状态、进度 | 状态变更 | 状态同步 |

##### 4.3.1.4 成本分摊

**建设诉求**：自动核算项目成本，支持单归属双标记分摊模式

**建设内容**：
- 人工成本自动分摊
- 项目成本归集
- 成本偏差分析
- 成本数据回流

**业务流程**：

```
成本归集规则：

【人员成本归集 - 单归属】
人工成本：人员(bu_id + bu_type)→工时→任务→WBS→项目
• 研发/交付人员：bu_type=事业部，成本归集到所属事业部
• 销售人员：bu_type=阿米巴，成本归集到所属阿米巴

【项目成本归集 - 双标记】
项目同时标记：bu_id(执行方) + amoeba_id(销售方)
• 交付成本 → 100%归集到 bu_id(事业部)
• 销售成本 → 100%归集到 amoeba_id(阿米巴)
• 项目利润 = 收入 - 交付成本 - 销售成本

成本分摊流程：
工时填报(人员+任务+工时) → 工时成本计算(按人员职级) → 归集到项目 
    → 按双标记分摊 → 生成成本报表 → 同步用友(直接回写每个项目投入人员明细：谁、多少时间、什么岗位)
```

| 伙伴云→用友 | 项目预算 | 预算生成 | API对接（施工预算、采购预算、运维预算、定制研发预算） |
| 伙伴云→用友 | 项目人员投入明细(人员+工时+岗位) | 实时/每日 | 定时同步，直接回写每个项目投入明细 |

#### 4.3.3 质量管控

质量管控子模块作为研发交付过程的保障环节，通过Bug管理、评审管理和质量分析确保交付质量。

##### 4.3.3.1 Bug管理

**建设诉求**：全生命周期缺陷管理，支持激活次数统计和质量分析

**建设内容**：
- Bug录入与分类
- 状态流转管理
- 激活次数统计
- 角色权限控制

**业务流程**：

```
Bug全生命周期流程：

新BUG(激活)
    ↓
已确认 ←──────┐
    ↓         │
已解决 ──→ 激活（激活次数+1）
    ↓
已关闭

状态流转规则：
• 新BUG（激活）→ 已确认/已拒绝（质控/项目管理确认Bug有效）
• 已确认 → 已解决/已拒绝（研发处理完成或判定不修复）
• 已解决 → 已关闭/激活（测试验证通过关闭，或验证不通过重新激活）
• 已关闭 → 激活（问题复发或关闭错误）
• 已拒绝 → 激活（拒绝理由不充分或误判）
```

**角色权限控制**：

| 角色 | 权限范围 | 特殊限制 |
|------|---------|---------|
| **质控角色** | 不受限制 | 不可标记Bug为"已解决"（只能标记为已确认/关闭/拒绝/激活） |
| **研发角色** | 常规操作 | 不可删除Bug，不可将Bug变更为"已关闭"（需测试人员关闭） |
| **项目管理角色** | 不受限制 | 拥有所有操作权限 |

**Bug表单字段**：

**基础信息**：

| 字段 | 类型 | 必填 | 说明 | 默认值 |
|------|------|------|------|--------|
| 所属产品线 | 下拉选择 | 是 | 自动填充，可修改 | 当前产品线 |
| 所属产品 | 下拉选择 | 是 | 与产品线联动筛选 | - |
| 所属功能 | 文本 | 否 | 功能模块名称 | - |
| 迭代版本 | 下拉选择 | 否 | 与产品联动，自动填充 | 最新版本 |
| 指派给 | 人员选择 | 是 | 指定处理人 | 自动分配 |
| Bug标题 | 文本 | 是 | 简要描述问题 | - |
| Bug内容 | 富文本 | 是 | 支持粘贴图片 | - |

**分类信息**：

| 字段 | 类型 | 选项 | 说明 |
|------|------|------|------|
| **Bug类型** | 单选 | 设计缺陷/前端代码错误/后端代码错误/功能缺失/性能问题/安装部署/配置相关/界面优化 | 分类统计 |
| **严重程度** | 单选 | S0致命/S1严重/S2一般/S3轻微 | 修复优先级 |
| **优先级** | 单选 | 高/中/低 | 处理优先级 |
| **Bug来源** | 单选 | 线上BUG/线下BUG | 线下BUG关联提测单 |
| **关联需求** | 选择 | - | 关联到具体需求 |
| **选择提测单** | 选择 | - | 线下BUG时关联提测记录 |

**严重程度定义**：

| 级别 | 定义 | 处理时效 | 示例 |
|------|------|---------|------|
| **S0致命** | 系统崩溃、数据丢失、核心功能完全不可用 | 2小时内响应，24小时内修复 | 系统无法登录、数据库崩溃 |
| **S1严重** | 主要功能异常、重要业务流程阻断 | 4小时内响应，48小时内修复 | 支付功能失败、关键页面报错 |
| **S2一般** | 次要功能异常、非核心流程问题 | 1个工作日内响应，版本内修复 | 界面显示异常、非关键功能缺陷 |
| **S3轻微** | 界面优化、交互体验改进 | 排期处理 | 文字错误、样式微调 |

**解决方案说明**：

| 解决方案 | 适用场景 | 后续处理 |
|---------|---------|---------|
| **已解决** | 正常修复完成 | 进入验证流程 |
| **设计如此** | 符合设计预期，非缺陷 | 需产品确认 |
| **重复BUG** | 与已有Bug重复 | 关联重复Bug |
| **外部原因** | 第三方系统/环境问题导致 | 记录外部原因 |
| **无法重现** | 无法复现问题 | 可标记观察 |
| **延期处理** | 暂时不修复，后续版本处理 | 记录延期原因 |
| **不予解决** | 确定不修复 | 需说明原因 |

**激活次数统计规则**：
- 激活次数从"已解决"状态被重新激活时计算
- 每次从"已解决" → "激活"，激活次数+1
- 用于衡量Bug修复质量和测试验证效果

**业务含义**：
- 激活次数=0：首次修复即通过
- 激活次数=1-2：修复质量一般，需关注
- 激活次数≥3：修复质量差，需复盘分析

**数据交互**：

无

#### 4.5.4 成本核算

**成本归集规则**：

```
【人员成本归集 - 单归属】
人工成本：人员(bu_id + bu_type)→工时→任务→WBS→项目
• 研发/交付人员：bu_type=事业部，成本归集到所属事业部
• 销售人员：bu_type=阿米巴，成本归集到所属阿米巴

【项目成本归集 - 双标记】
项目同时标记：bu_id(执行方) + amoeba_id(销售方)
• 交付成本 → 归集到 bu_id(事业部)
• 销售成本 → 归集到 amoeba_id(阿米巴)
• 项目利润 = 收入 - 交付成本 - 销售成本

【成本分摊比例】
• 交付相关成本：100% 归集到 bu_id(事业部)
• 销售相关成本：100% 归集到 amoeba_id(阿米巴)
• 公共成本：可按收入比例或协商比例分摊
```

**数据交互**：

| 接口方向 | 数据内容 | 触发时机 | 实现方式 | 说明 |
|---------|---------|---------|---------|------|
| **伙伴云→用友** | 人工成本、工时 | 每日/周 | 定时同步 | 用于成本核算 |
| **用友→伙伴云** | 成本核算结果 | 月末 | 定时同步 | 实际成本回流展示 |
| **用友→伙伴云** | 确认收入 | 财务记账 | 定时同步 | 用于利润分析 |

**成本回流展示**：
- 用友核算 → 伙伴云展示
- 展示维度：预算vs实际
- 展示视图：按事业部和阿米巴分别展示
