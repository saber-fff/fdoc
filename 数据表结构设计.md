**业务单元类型说明(并存模式)**:

根据2026组织架构变革，采用**事业部和阿米巴并存**模式：

| 架构模式 | 业务单元类型 | 典型单元 | 适用场景 |
|---------|-------------|---------|---------|
| **事业部制** | 事业部 | 大健康运营事业部<br>标准化项目事业部<br>大项目事业部<br>效能与品质保障事业部 | 研发/交付/平台 | 产品研发、标准化交付、平台支撑 |
| **阿米巴制** | 经营部 | 经营一部<br>经营二部<br>经营三部<br>经营四部 | 销售 | 区域化项目销售、客户关系 |

**并存架构核心原则**：
1. **架构并存**：事业部和阿米巴同时存在，不是替代关系
2. **人员单归属**：每个人员只归属一种架构
   - 研发/交付/平台人员 → 归属事业部（职能线）
   - 销售人员 → 归属阿米巴（销售线）
3. **项目双标记**：项目同时标记
   - `bu_id`：执行方（事业部，负责交付）
   - `amoeba_id`：销售方（阿米巴，负责销售）
4. **成本分开核算**：按人员归属架构分别核算成本

**联系人类型说明**：

| 类型 | 说明 | 维护要求 |
|------|------|----------|
| **关键人** | 对项目决策有重要影响的人 | 必须维护，定期跟进 |
| **经办人** | 具体业务经办人员 | 必须维护 |
| **其他** | 一般联系人 | 按需维护 |

**决策角色说明**：

| 角色 | 职责 | 沟通策略 |
|------|------|----------|
| **决策者** | 有最终决策权 | 高层对接，战略层面沟通 |
| **影响者** | 影响决策但无最终决定权 | 技术/业务层面深入沟通 |
| **执行者** | 负责具体执行 | 操作层面支持，日常维护 |

**合同外功能业务说明**：

| 业务类型 | 说明 | 成本记录 | 关联字段 |
|---------|------|---------|---------|
| **赠送** | 免费提供给客户的功能 | ✅记录成本 | cost_amount |
| **置换** | 客户用其他合同中的功能置换 | ✅记录成本 | ref_contract_id(来源合同) |
| **补签** | 先实施后补签合同 | ✅记录成本 | target_contract_id(补签后合同) |
| **提前施工** | 合同签订前提前实施 | ✅记录成本 | 后续关联正式合同 |
| **其他** | 其他合同外工作 | ✅记录成本 | 自定义说明 |

```

**版本提测检查**：

| 检查项 | 检查逻辑 | 检查结果 |
|--------|---------|---------|
| 需求评审 | 所有关联需求的需求评审状态 | 必须全部"已通过" |
| 接口评审 | 所有关联需求的接口评审状态 | 必须全部"已通过" |
| 设计评审 | 需要设计的需求的UI评审状态 | 必须全部"已通过" |
| 用例评审 | 需要测试用例的需求的用例评审状态 | 必须全部"已通过" |
| 演示评审 | 需要演示的需求的演示评审状态 | 必须全部"已通过" |

检查不通过时，系统提示哪些需求的哪些评审未完成，不能发起提测。

**同步状态说明**：

| sync_status | 说明 | 处理建议 |
|-------------|------|---------|
| 未同步 | 伙伴云已创建，未同步内网 | 系统自动触发同步 |
| 同步中 | 正在同步到内网系统 | 等待完成 |
| 已同步 | 内网系统已创建对应版本 | 正常状态 |
| 同步失败 | 同步过程中发生错误 | 查看sync_error，手动重试 |

**同步流程**：
```
创建版本 → sync_status=未同步 → 调用内网API → sync_status=同步中
                                              ↓
                                    成功: sync_status=已同步
                                    失败: sync_status=同步失败
                                          sync_error=错误详情
```

|---------|---------|---------|--------|---------|---------|---------|---------|
| 需求评审 | default | always | - | 需求评审 | 评审 | 产品部 | 4 |
| 接口设计 | default | always | - | 接口设计 | 设计 | 研发部 | 8 |
| 接口评审 | default | always | - | 接口评审 | 评审 | 架构组 | 2 |
| 用例设计 | estimate_days | > | 3 | 用例设计 | 设计 | 测试部 | 6 |
| 用例评审 | estimate_days | > | 3 | 用例评审 | 评审 | 测试部 | 2 |
| UI设计 | has_new_page | is_true | - | UI设计 | 设计 | 设计部 | 12 |
| UI评审 | has_new_page | is_true | - | UI评审 | 评审 | 设计部 | 2 |
| 演示评审 | need_demo | is_true | - | 演示评审 | 评审 | 产品部 | 2 |
| 试点任务 | need_pilot | is_true | - | 试点 | 执行 | 交付部 | 16 |

**任务生成逻辑**:

```
需求审核通过
    ↓
遍历所有启用的任务配置(dev_task_config)
    ↓
检查触发条件：
├── default类型 → 必生成
├── estimate_days > 3 → 当预估时间>3天时生成
├── has_new_page = true → 当有新页面时生成
├── need_demo = true → 当需要演示时生成
└── need_pilot = true → 当需要试点时生成
    ↓
生成任务(dev_task)：
├── task_name = 任务类型 + "-" + 需求标题
├── task_type = 配置中的任务类型
├── dept_id = 配置中的处理部门
├── plan_hours = 配置中的标准工时
├── priority = 继承需求优先级
├── status = 待分配
└── create_time = 当前时间
    ↓
部门负责人二次分配给具体执行人
```

| 类型值 | 说明 | 适用场景 |
|--------|------|----------|
| 设计缺陷 | 设计层面的问题 | 交互逻辑不合理、业务流程缺陷 |
| 前端代码错误 | 前端实现问题 | UI显示异常、前端逻辑错误 |
| 后端代码错误 | 后端实现问题 | API错误、数据处理异常 |
| 功能缺失 | 功能不完整 | 缺少必要功能、功能未实现 |
| 性能问题 | 性能相关 | 响应慢、内存泄漏、卡顿 |
| 安装部署 | 部署问题 | 安装失败、配置错误 |
| 配置相关 | 配置问题 | 参数配置错误、环境配置问题 |
| 界面优化 | 界面改进 | 样式调整、布局优化 |

**严重程度枚举**：

| 级别 | 值 | 定义 | 处理时效 |
|------|-----|------|----------|
| **S0致命** | S0 | 系统崩溃、数据丢失、核心功能完全不可用 | 2小时内响应，24小时内修复 |
| **S1严重** | S1 | 主要功能异常、重要业务流程阻断 | 4小时内响应，48小时内修复 |
| **S2一般** | S2 | 次要功能异常、非核心流程问题 | 1个工作日内响应，版本内修复 |
| **S3轻微** | S3 | 界面优化、交互体验改进 | 排期处理 |

**Bug来源枚举**：

| 来源 | 说明 | 关联要求 |
|------|------|----------|
| **线上BUG** | 生产环境发现的Bug | 无需关联提测单 |
| **线下BUG** | 测试环境发现的Bug | 必须关联提测单(test_request_id) |

**状态流转规则**：

| 当前状态 | 可变更状态 | 操作权限 |
|---------|-----------|----------|
| 新BUG(激活) | 已确认、已拒绝 | 质控、项目管理 |
| 已确认 | 已解决、已拒绝 | 研发、项目管理 |
| 已解决 | 已关闭、激活 | 质控(关闭)、测试(激活) |
| 已关闭 | 激活 | 质控、项目管理 |
| 已拒绝 | 激活 | 质控、项目管理 |

**解决方案枚举**：

| 解决方案 | 适用场景 | 后续处理 |
|---------|---------|----------|
| **已解决** | 正常修复完成 | 进入验证流程 |
| **设计如此** | 符合设计预期，非缺陷 | 需产品确认 |
| **重复BUG** | 与已有Bug重复 | 关联到原Bug(duplicate_bug_id) |
| **外部原因** | 第三方系统/环境问题导致 | 记录外部原因 |
| **无法重现** | 无法复现问题 | 可标记观察 |
| **延期处理** | 暂时不修复，后续版本处理 | 记录延期原因 |
| **不予解决** | 确定不修复 | 需说明原因 |

**激活次数计算规则**：

- 每当状态从"已解决"变更为"激活"时，activate_count + 1
- 用于衡量Bug修复质量和测试验证效果
- 激活次数≥3的Bug需要质量复盘分析

## 1.5 组织架构数据模型说明

### 2026年组织架构变革

根据2026年组织架构调整，从**一线阿米巴**向**事业部制**转型：

#### 架构对比

| 维度 | 一线阿米巴(旧) | 事业部制(新) |
|------|---------------|-------------|
| **组织层级** | 经营部→区域组 | 事业部→职能部门 |
| **划分依据** | 地理区域(中部/西部/南部等) | 业务类型(运营/标准化/大项目) |
| **典型单元** | 经营一部→中部A2组 | 大健康运营事业部→研发部 |
| **人员归属** | 单归属(销售→阿米巴) | 单归属(研发/交付/平台→事业部) |

#### 数据模型设计

```
┌─────────────────────────────────────────────────────────────────┐
│                    组织架构数据模型                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   sys_business_unit (业务单元) - 同时支持两种架构               │
│   ├── 事业部类型:                                              │
│   │   ├── 大健康运营事业部 (biz_type=运营)                      │

│   │   ├── 大项目事业部 (biz_type=大项目)                        │
│   │   └── 效能与品质保障事业部 (biz_type=平台)                  │
│   │                                                            │
│   └── 经营部类型(并存):                                        │
│       ├── 经营一部 (region_scope=[中部])                        │
│       ├── 经营二部 (region_scope=[中部])                        │
│       ├── 经营三部 (region_scope=[西部,南部,东部])              │
│       └── 经营四部 (region_scope=[北部,西部])                   │
│                                                                │
│   sys_dept (部门) - 隶属于业务单元                             │
│   ├── 事业部模式下:                                            │
│   │   ├── 研发部 (dept_type=研发部)                            │
│   │   ├── 交付部 (dept_type=交付部)                            │
│   │   ├── 运营部 (dept_type=运营部)                            │
│   │   ├── 测试质控部 (dept_type=测试质控部)                    │
│   │   ├── 研发平台部 (dept_type=研发平台部)                    │
│   │   └── 工程效能部 (dept_type=工程效能部)                    │
│   │                                                            │
│   └── 经营部模式下:                                            │

│       ├── 西部2组 (dept_type=区域组, region_code=西部)         │
│       └── ...                                                  │
│                                                                │
│   sys_user (人员) - 单归属                                     │
│   ├── bu_id (所属业务单元) - 用于成本归集                      │
│   ├── bu_type (架构类型) - 区分事业部/阿米巴                   │
│   └── dept_id (所属部门) - 用于日常管理                        │
│                                                                │
│   【人员归属流向】                                             │
│   研发/交付/平台人员 → 归属事业部 (bu_type='事业部')          │

│                                                                │
└─────────────────────────────────────────────────────────────────┘
```

#### 成本核算规则

1. **人工成本归集**: 按照 `user.bu_id` + `user.bu_type` 归集到所属架构
   - 研发/交付/平台人员成本 → 归集到事业部
   - 销售人员成本 → 归集到阿米巴
2. **项目成本**: 项目同时标记 `bu_id`(执行方-事业部) 和 `amoeba_id`(销售方-阿米巴)
   - 交付相关成本 → 归集到 bu_id(事业部)
   - 销售相关成本 → 归集到 amoeba_id(阿米巴)

**数据校验规则**:

| 校验项 | 规则 | 错误提示 |
|-------|------|---------|
| 工时范围 | 0.5 ≤ hours ≤ 12 | 单日工时必须在0.5-12小时之间 |

| 单日上限 | 同一用户同日总和 ≤ 12 | 该日期已填报XX小时，超出单日上限 |

```

场景3：跨事业部协作（业务B执行业务A的任务）
yearly_goal [bu_id=业务A]
    → monthly_task [bu_id=业务A, source_bu_id=业务A]
        → weekly_task [bu_id=业务B, source_bu_id=业务A] ★

**备注**:
1. 所有表均需包含 create_time, update_time, create_by, update_by 字段
2. 删除采用逻辑删除(is_deleted字段)，保留数据完整性
3. 大文本字段(description, content等)建议单独存储或使用压缩
4. 财务相关数据需与用友系统保持一致性校验
5. **组织架构并存期**: 2026年事业部和阿米巴并存，通过`bu_type`字段区分
6. **人员单归属**: 人员仅归属一种架构（bu_id + bu_type），研发/交付/平台→事业部，销售→阿米巴
7. **项目双标记**: 项目同时标记bu_id(执行方-事业部)和amoeba_id(销售方-阿米巴)
8. **区域字段**: 经营部模式下使用region_code标记地理区域，用于区域化管理和成本核算
9. **任务层级管理**: 年度目标→月任务→周任务→每日工时，支撑事业部工时分摊通过source_bu_id实现
10. **预算管理**: 伙伴云负责预算编制(fin_budget_*)，用友负责预算执行监控和预警
11. **采购管理**: 伙伴云发起采购申请(pur_request)，用友执行采购流程并回传订单状态(pur_order)
12. **应收应付**: 伙伴云展示应收应付数据(fin_ar/fin_ap)，用友负责实际收付款和账务处理
13. **成本归集**: 工时成本自动归集到项目，按bu_id和amoeba_id双标记分摊，实时同步用友
14. **数据流向**: 伙伴云→用友(预算/申请/工时)，用友→伙伴云(执行/回款/订单)
